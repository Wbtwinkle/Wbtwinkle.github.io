Web会话管理机制-Cookie、Session、JWT

cookie session jwt
Network
发布日期:   2022-01-08

Web 开发模式

服务端渲染

优点：

  1. 前端耗时少，服务器动态生成HTML内容，浏览器只需要请求服务器获取页面即可
  2. 有利于SEO优化，服务器端响应的是完整的HTML页面内容，利于SEO

缺点：

  1. 占用大量服务器资源，因为在服务器端完成页面内容的拼接，如果请求较多，服务器负载会较大
  2. 不利于前后端分离，开发效率低，使用服务器端渲染，则无法分工合作，尤其对于前端复杂度高的项目，开发效率低。
前后端分离

依赖于 Ajax 技术的广泛应用，后端只需要提供 API 接口，前端使用网络请求技术例如 Ajax 或者第三方框架 Axios，调用接口以获取数据，进行页面渲染。

优点：

  1. 分工明确，前端专注于 UI 页面的开发，后端专注于 API 的开发
  2. 利于优化用户体验，Ajax 技术可以实现局部页面刷新
  3. 减轻了服务器端的渲染压力

缺点：

  1. 不利于SEO，但可以通过 Vue、React 框架等的 SSR 技术解决此问题。

身份认证

不同开发模式下的认证方式

服务端渲染推荐使用 Session 认证机制，前后端分离推荐使用 JWT 认证机制

HTTP无状态性

在HTTP 1.x版本协议中，HTTP是无状态协议，也就是说，当客户端发送给服务器端请求时，服务器端不会保存客户端的身份，当相同的客户端下一次再次访问服务器端时，服务器端无法识别是否是同一个客户端发送的请求。

Cookie

参考：https://juejin.cn/post/6844904095711494151

使用Cookie可以解决HTTP无状态导致的问题，它是存储在用户本地的小型文本数据。一般不超过4KB。

如何设置Cookie？Cookie 在身份认证中的作用？

首先用户在客户端给服务器端发送一个请求，服务器端接收到客户端的请求后，生成一个Cookie，在响应中添加Set-Cookie字段，返回给客户端。当用户接收到来自服务器端的响应之后，会把Cookie保存在本地，在以后请求这个服务器时，会自动携带上本地未过期的Cookie，发送给服务器。

Cookie的属性：

  1. expires 过期时间：它限制了在何时保存在本地浏览器或者硬盘中的Cookie会失效，当这个值未被指定时，默认为会话性Cookie【值为Session】，而会话性Cookie指的就是这个Cookie值保存在了客户端浏览器中，当用户关闭浏览器即失效。

    有些浏览器提供会话恢复功能，即使关闭了浏览器，会话Cookie也会保留下来。与会话Cookie相对应的是持久性Cookie，持久性的Cookie会保存在用户本地硬盘中，直到指定expires过期时间才会失效。

    值得注意的是，expires 它是HTTP 1.0协议的字段，在HTTP 1.1协议中expires被Max-Age替代。{ 这貌似是因为 expires 它的过期时间是客户端的，然而客户端和服务器端的时区可能不同，会造成意外失效？} Max-age就不会有这种意外，因为Max-age是以设置Cookie后多少秒以后失效，不是一个绝对值，是一个相对值。

    假如浏览器中 expires 字段和 Max-Age 同时存在，那么Max-Age优先级更高，如果Max-Age的值是一个正数，那么会持久化保存，如果是负数，相当于是会话级别的Cookie，如果是0，代表立即失效。

  2. Domain 域：它可以限制客户端在请求哪些服务器时，才可以携带Cookie。如果客户端要访问的域名是Cookie中Domain的值，或者是这个值的子域，那么就会携带Cookie信息进行请求。

    比如Cookie中保存的Domain值为 baidu.com，那么我们访问baidu.com时或者访问百度手机端m.baidu.com时都会携带Cookie信息。

  3. Path 路径：它和Domain的作用类似，Domain是限制域名，而Path是限制路径，只有是Path指定的路径或者其子路径才可以携带Cookie发送请求。

  4. Secure 安全标志：它是唯一一个不需要键值对形式的Cookie属性，当使用了这个secure标志之后，只有经过HTTPS SSL安全连接的情况下，才可以发送Cookie；也就是HTTP协议不会携带Cookie，HTTPS协议才可以携带Cookie，它可以保护Cookie在网络传输过程中的安全。

  5. HttpOnly：设置了这个属性为true后，防止客户端脚本通过document.cookie等方式访问Cookie，可以避免XSS攻击

  6. SameSite：2020年2月份发布的Chrome 80版本，默认屏蔽了第三方Cookie，SameSite属性值由旧版本的none改为了默认为Lax。当属性值为none时，无论是否跨站，都会发送Cookie；当属性值为Lax时，允许部分第三方请求发送Cookie，使用post表单、Ajax、iframe、Image src属性都不会再发送Cookie.【需要进一步了解，个人目前没有搞的特别清楚】

Cookie 的缺陷
  1. 存在限制，每个域最多20个Cookie，每个Cookie最多4KB内存，超出域限制的Cookie会被自动清除
  2. 部分Cookie不需要频繁发送，但是每次请求都会携带上所有Cookie
  3. 存在安全隐患，如果没有设置HttpOnly的话，会有XSS攻击风险，容易伪造Cookie
Cookie 的作用
  1. 会话状态管理（用户登录状态、购物车不会刷新一次就被清空等）
  2. 个性化设置（识别同一个客户端用户的身份，保留用户自定义设置等）
  3. 浏览器行为跟踪（分析用户行为）
Session

Session是另一种记录会话状态的机制，Session是基于Cookie实现的，它存储于服务器端。

具体流程是：当用户在客户端第一次请求服务器时，服务器根据用户信息创建Session，保存在服务器内部，并在响应中发送给客户端一个SessionID作为客户端标识信息。浏览器接收到服务器返回的SessionID信息后，将信息存入到Cookie中。当下一次用户访问服务器时，会携带Cookie发送给服务器端，服务器从Cookie中获取SessionID查找服务器内存中的对应Session，如果找到，则用户可继续执行后续操作，若未找到，则说明用户信息失效。

Cookie 与 Session 对比

安全性：Session 比 Cookie 安全，Session 存储在服务器端，Cookie 存储在本地，可以被伪造修改。

存取值的类型：Cookie只支持字符串型数据，Session可以存放任意类型数据。

有效期：Cookie可设置为长时间保存，Session存储在服务器上，会对服务器造成一定压力。默认情况下客户端关闭就会失效，也不建议将其长时间存储。

存储大小：Cookie不能超过4KB，Session远远大于Cookie，但是会占用服务器资源。

JWT 认证机制

可以参考一下：http://www.ruanyifeng.com/blog/2018/07/json_web_token-tutorial.html

JWT全称 JSON Web Token，是目前最流行的跨域认证解决方案。由于Session需要和Cookie搭配使用，而Cookie无法跨域，所以需要额外进行很多配置实现跨域，因此在跨域情境下实现身份认证通常采用JWT方式。

JWT 工作原理

用户在浏览器端向服务器端发送请求，服务器端接收用户请求，验证用户账号密码等，将用户信息加密成Token字符串，但不会保留在服务器端，而是响应回客户端。客户端接收到加密Token信息后存储到本地localStorage或SessionStorage，在用户下一次发送请求时，携带此加密信息于请求头Authorization中，一并发送给服务器端，服务器端解析请求头字段获取用户信息，根据得到的用户信息，针对不用用户响应不同内容给用户客户端浏览器。

JWT 组成部分

JWT由三部分组成，分别为Header、Payload以及Signature，用户信息加密后就形成了Payload部分，而Header和Signature部分是安全性相关部分，作用是保证Token安全性。

JWT 的使用

在Express框架中，使用JWT，需要安装两个JWT相关第三方包。

npm install jsonwebtoken npm install express-jwt

jsonwebtoken

用于生成 JWT 加密字符串

express-jwt

用于将 JWT 加密字符串解析还原成 json 对象格式信息

secret秘钥

为了保证 JWT 字符串的安全性，防止 JWT 字符串在网络传输过程中被别人破解，我们需要专门定义一个用于加密和解密的 secret 密钥，生成 JWT 字符串时，使用秘钥加密，解析 JWT 字符串为 JSON 对象时，需要使用秘钥解密。
